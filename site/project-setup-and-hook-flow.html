Training Wheels / Architecture

Project setup and the hook evaluation path

Training Wheels is a Go CLI that sits in front of agent-issued shell commands. An agent calls the installed hook, the hook turns protocol-specific JSON into one shell command string, and the evaluator decides whether the command is safe to run, should be blocked, or should require confirmation.

Primary binary cmd/tw -> internal/cli
Main hook entry tw hook reads JSON on stdin
Core decision engine internal/eval.Evaluator

Mental Model

The easiest way to think about the project is as five layers. The CLI accepts commands, the hook layer understands agent payloads, the app layer assembles runtime dependencies, the evaluator runs the decision pipeline, and the pack/rule/override/session data determines the result.

1. Interface layer

cmd/tw/main.go and internal/cli expose commands like hook, test, explain, install, and rule.

2. Hook protocol layer

internal/hook normalizes Claude, Copilot, and Gemini hook payloads into one extracted command plus a detected protocol.

3. Runtime assembly

internal/app loads config, packs, rules, overrides, and session state and builds an evaluator with the right shell context.

4. Evaluation engine

internal/eval, internal/ast, and internal/normalize implement the decision pipeline and structural matching.

5. Policy data

Built-in packs live under packs/json. User and project policy comes from ~/.tw and .tw files.

6. Install surface

internal/cli/install.go, internal/agentsettings, and internal/skills wire TW into supported agent settings.

Repo Map

These are the directories that matter most when you are tracing runtime behavior. Most other directories are tests, docs, release plumbing, or labs for experimentation.

Path What it owns Why it matters for hook flow

internal/cli Cobra commands, flags, install/uninstall, explain/test subcommands. internal/cli/hook.go is the concrete runtime path for agent hook execution.

internal/override, internal/rules, internal/session User/project exceptions and temporary approvals. These can allow or ask before pack evaluation, and can also override a destructive pack match after the fact.
packs/json Built-in policy data shipped with the binary. Loaded into the default registry and turned into active packs for evaluation.

Runtime Setup

Before the hook can evaluate anything, TW has to be installed into an agent's settings and the binary has to know where to find its policy inputs.

Install path

tw install copies the current binary to a stable path under ~/.tw/bin, installs embedded skills, detects supported agent settings, and injects the hook reference there. That logic lives in internal/cli/install.go and internal/agentsettings.

Registry setup

internal/app/runtime.go calls EnsureDefaultRegistry(), which registers built-in packs from embedded JSON via internal/packs/allpacks/allpacks.go and then loads any configured external pack files or directories.

Built-in packs embedded from packs/json/*.json User config at ~/.tw/config.json External pack roots: ~/.tw/packs and .tw/packs User/project rules at ~/.tw/rules.json and .tw/rules.json User/project overrides at ~/.tw/overrides.json and .tw/overrides.json Session token and allowlist under ~/.tw/
Important setup detail

TW is intentionally fail-open on hook runtime problems. If config loading, hook input parsing, command extraction, or deadline budget handling fails inside tw hook, the command is allowed rather than blocked.

Hook Ingress Flow

This is the path from “the agent wants to run a command” to “the evaluator has one command string to inspect.”

Agent calls the installed hook

The agent runtime invokes tw hook as a pre-execution hook and sends a JSON payload over stdin. Supported protocols are Claude, Copilot, and Gemini.

CLI command dispatch enters runHook

cmd/tw/main.go calls cli.Execute(). Cobra resolves the hook subcommand and runs runHook in internal/cli/hook.go.

Config is loaded first

app.LoadConfig() resolves defaults, user config, and env overrides. This establishes limits like hook_timeout_ms, max_command_bytes, active packs, severity threshold, and default action.

Hook payload is decoded

hook.ReadHookInput() reads bounded stdin bytes and unmarshals a union struct that accepts both snake_case and camelCase fields for protocol differences.

Protocol and command are extracted

hook.ExtractCommand() detects which protocol sent the payload and extracts a shell command only for shell-like tools such as bash, launch-process, or run_shell_command. Non-shell tool calls short-circuit to allow.

Shell context is inferred

runHook chooses a shell using the tool name first, then command-based detection, then a default shell. That shell choice controls whether AST parsing is enabled and which grammar to use.

Evaluator is built with optional state

app.NewEvaluator() loads overrides, custom rules, and session state on top of the enabled pack set. It also attaches the selected shell and timeout context.

The evaluator receives exactly one command string

From this point forward the pipeline operates on the original command plus a pre-parse-cleaned version. The result comes back as allow, deny, or ask along with metadata about the source of the match.

Hook output is written in the originating protocol

internal/hook/output.go formats protocol-specific JSON. Claude gets hookSpecificOutput, Copilot gets continue: false, and Gemini gets a decision field. Exit code handling also differs slightly by protocol.

Claude-style payload

{
"tool_name": "Bash",
"tool_input": {
  "command": "git reset --hard"
}
}

Copilot-style payload

{
"toolName": "launch-process",
"toolArgs": {
  "command": "rm -rf /"
}
}

Gemini-style shape

{
"event": "BeforeTool",
"session_id": "...",
"tool_name": "run_shell_command",
"tool_args": {
  "command": "git status"
}
}

Decision Pipeline

The evaluator is the heart of the system. The code in internal/eval/evaluator.go is sequential on purpose: it performs cheap allow/ask checks first, prunes irrelevant packs, then spends the parse budget only where it is needed.

Empty command check

Blank or whitespace-only commands immediately return allow.

Pre-parse cleanup

internal/normalize/normalize.go strips carriage returns, collapses line continuations, and normalizes Windows absolute path backslashes so parsing behaves consistently.

Pre-pack custom rule allows

allow rules are checked before pack evaluation. Command-kind allow rules can parse the command structurally and bypass all later pack checks.

Pre-pack override ask

Session/time-scoped overrides from user or project state can require confirmation before any pack logic runs.

Pre-pack override allow

Exact, prefix, or rule-targeted overrides can allow the command early.

Pre-pack session allow or ask

HMAC-protected session entries are checked next. These are ephemeral approvals stored under ~/.tw/.

Quick-reject keyword scan

internal/eval/quick_reject.go uses Aho-Corasick over enabled pack keywords. If no relevant keywords match, the evaluator returns allow without doing AST work.

AST parse and command enrichment

If AST mode is enabled for the current shell, TW parses with tree-sitter, unwraps wrappers, collects simple commands, and enriches CLI subcommands for tools like git, docker, and kubectl.

Deadline budget check

The evaluator respects the hook timeout context. If the budget is exhausted, it returns allow with SkippedDueToBudget=true.

Structural pack evaluation

Only candidate packs are scanned. A pack matches by checking parsed SimpleCommand values against structural when/unless patterns, not raw shell text.

Post-match exception handling

Once a pack finds a destructive match, TW runs a second round of checks: allow rules by rule ID, override ask, override allow, and session allow/ask. This is where “allow this exact matched rule” works.

Final decision

If nothing overrides the match, the pack action wins: usually deny, sometimes ask. If no pack matches, the result is allow. Back in runHook, a pack denial can be upgraded to ask when config packs.default_action is ask.

allow: command runs silently or with a logged allow source such as a custom rule, override, or session entry.
ask: hook returns structured output telling the agent the command needs review/confirmation.
deny: hook returns a blocking payload containing reason, rule ID, pack ID, and severity when available.
Why the AST matters

The evaluator is designed to understand command structure rather than just scanning text. That is why destructive strings inside data positions, such as a commit message, can remain allowed while real command invocations are blocked.

State and Data Sources

The decision result is a merge of shipped policy and local state. If you are trying to explain a surprising allow or deny, these are the files to inspect first.

Source Location Effect on evaluation
Built-in packs packs/json/*.json Define the baseline destructive command patterns and their keywords.
External packs ~/.tw/packs, .tw/packs, plus configured paths Add more packs to the registry at startup.
Config ~/.tw/config.json plus env vars Controls time budget, max input size, pack enablement, severity threshold, and default action.
Persistent rules ~/.tw/rules.json, .tw/rules.json Allow rules are checked directly; deny and ask rules are converted into a synthetic pack named rules.custom.
Overrides ~/.tw/overrides.json, .tw/overrides.json Provide exact, prefix, or rule-based ask/allow entries with project precedence over user scope.
Session allowlist ~/.tw/session-token, ~/.tw/allow.key, ~/.tw/allow-<token-prefix>.json Provides ephemeral allow or ask entries that are signed to prevent tampering.
Subtle but important ordering rule

Custom allow rules are not converted into pack denies. They are checked both before pack evaluation and after a pack match. Custom deny and ask rules, by contrast, are converted into a synthetic pack so they participate in normal structural matching.

Built-in Packs

The binary ships 110 built-in packs from packs/json/*.json. Those are the full policy catalog available to the registry, even though only a smaller subset is enabled by default in config.

Default vs shipped

By default, TW enables the core protection packs core.git, core.filesystem, and core.tw, plus windows on Windows. The rest of the built-in packs are still shipped and can be enabled through config or pack selection.

Start Here in Code

If you want to keep tracing this manually, this order gives the shortest path through the runtime.

cmd/tw/main.go

Confirms the entire binary is just CLI dispatch plus exit code handling.

internal/cli/hook.go

Best single file for the top-level hook lifecycle: config, input, extraction, evaluator construction, and protocol-specific output.

internal/hook/input.go and internal/hook/output.go

Use these to understand payload shapes, protocol detection, and emitted deny/ask JSON.

internal/app/runtime.go

This is the assembly layer that turns config and local state into a ready evaluator.

internal/eval/evaluator.go and internal/eval/quick_reject.go

Read these for the authoritative decision ordering and the cheap prefilter that keeps the hook fast.

internal/ast and packs/json

These explain what “structural matching” really means and what patterns are actually shipped.

Fastest debugging command

Run tw explain "your command here" to see the traced evaluation steps, the normalized command, the matched pack or rule, and the final decision without going through a real agent hook invocation.
